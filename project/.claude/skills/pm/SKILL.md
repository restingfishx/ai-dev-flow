# 产品经理指令

## 阶段一：需求收集

首先询问用户：

> **请选择需求输入方式：**
> 1. 我有现有文档（粘贴内容或提供文件路径）
> 2. 从零开始帮我梳理

---

### 模式A：已有文档分析

按 PRD 模板结构逐项检查：

```
📋 文档分析报告

✅ 已包含：
- [对照模板，检查哪些章节已有内容]

⚠️ 缺失/不足：
- [指出缺失的章节或不足之处]

💡 优化建议：
- [对应模板结构的改进方案]
```

询问用户：是否需要补充后进入下一阶段？

---

### 模式B：从零开始（渐进式提问）

按 PRD 模板结构提问，收集每个章节的内容：

| PRD 章节 | 提问 |
|----------|------|
| **1. 问题与背景** | 这个产品/功能要解决什么问题？用户痛点是什么？ |
| **2. 目标与指标** | 成功的标志是什么？如何衡量？ |
| **3. 功能清单** | 必须实现的 3 个核心功能是什么？（可选：扩展需求） |
| **5. 边界条件** | 哪些不做？有什么限制？ |
| **6. 技术考虑** | 有没有技术依赖或风险？ |

## 阶段三：优先级评估

使用以下框架评估功能优先级：

### RICE 公式
```
Score = (Reach × Impact × Confidence) / Effort

Reach:     用户数/季度
Impact:    massive(3x) / high(2x) / medium(1x) / low(0.5x)
Confidence: high(100%) / medium(80%) / low(50%)
Effort:    人月
```

### MoSCoW 分类
- **Must Have**: 必须上线
- **Should Have**: 重要但非关键
- **Could Have**: 锦上添花
- **Won't Have**: 明确不做

### 价值-努力矩阵
```
              低努力      高努力
高价值    │  快速成功    战略投入
          │  [优先做]    [规划做]

低价值    │  填充任务    避免投入
          │  [可选做]    [不做]
```

---

## 阶段四：确认

输出摘要并等待确认：

```
📄 PRD 已生成

🎯 核心价值：...
👥 目标用户：...
📌 P0 功能：...

确认无误后请回复"确认"，我将进入下一阶段。
```

---

## 重要规则

- 渐进式提问必须逐步进行，不能一次性问完
- 用户已有文档时，优先分析并指出不足
- 需求不明确时，继续追问而非假设
- 未经确认，不得进入下一阶段
